# TS Neighborhood Quality Improvement Plan

> Suggested file path: `docs/superpowers/plans/2026-06-21-ts-neighborhood-quality.md`

## Summary

**Goal:** Do not add or run any polish. Improve TS quality only by changing TS neighborhood candidate generation/selection, then benchmark whether TS no longer gets stuck at `90+`.

**Baseline:** commit `ab02e1a`.

| Algorithm | 300s Result on `R_30_10_2`, seed `44` |
|---|---:|
| ALNS 4000 iters | `65.72` |
| TS 300s | `92.62`, `27965` iterations |
| GA 300s | `73.20`, `1029` generations |

**Target:** TS should improve below `90.0` on the same 300s benchmark. GA remains unchanged and is measured only as a comparison baseline.

---

## Key Changes

### 1. Disable Polish Scope Explicitly

Do not add or call:

- `DroneTaskSplitMergeLocalSearch`
- `MultiCustomerSortieConstructor`
- `MatheuristicLNSRepair`
- `TruckDroneRechainRepair`

For this plan, remove or bypass any TS quality experiment path that relies on these operators after initial solution construction, during improvement, restart, or final return.

### 2. Replace TS Single Distance Prefilter With Mixed Neighborhood Buckets

Modify `heuristics/tabu_search/tabu_search.py`.

Current issue: TS evaluates many iterations but ranks candidates mostly by cheap distance proxy, so it keeps exploring locally short moves and misses drone-structure moves.

Implement bucketed candidate selection before full `_penalized_cost()` scoring:

| Bucket | Share | Scoring rule |
|---|---:|---|
| `distance_proxy` | 35% | current cheap solution distance proxy |
| `drone_saving` | 30% | prioritize moves that remove truck customers into drone tasks or improve drone/truck distance balance |
| `violation_fix` | 20% | prioritize moves generated by violation-directed generators |
| `random_diversity` | 15% | uniformly sample remaining candidates after deduplication |

Keep the existing downstream logic unchanged:

- tabu check
- `SearchEvaluator.verify_candidate`
- max drone task cap
- `_penalized_cost`
- best feasible tracking
- final feasible return fallback

Add configurable attributes in `TabuSearch.__init__`:

```python
self._candidate_bucket_enabled = True
self._candidate_bucket_shares = {
    "distance_proxy": 0.35,
    "drone_saving": 0.30,
    "violation_fix": 0.20,
    "random_diversity": 0.15,
}
```

Use these only for selecting which candidates enter expensive scoring.

### 3. Add TS Neighborhood Metrics

Extend TS stats with neighborhood diagnostics:

```python
self.stats["candidate_pool_size"]
self.stats["selected_candidate_count"]
self.stats["selected_by_bucket"]
self.stats["best_feasible_cost"]
```

Record per iteration:

- total generated candidates
- selected candidates after bucket filtering
- selected count by bucket
- current best feasible cost

This is required to explain whether TS is still stuck because of candidate generation or candidate acceptance.

### 4. Benchmark Script

Create or update a benchmark script, e.g. `scripts/benchmark_ts_neighborhood_quality.py`.

Required command:

```bash

  --instance R_30_10_2 \
  --size 10 \
  --seeds 44 \
  --time-limit 300 \
  --out results/ts_neighborhood_quality_R_30_10_2_seed44_300s.json
```

Output JSON must include:

- ALNS target cost

- GA final cost, feasibility, runtime, generations completed
- first hit for ALNS exact, ALNS+1%, ALNS+5%
- first hit for `cost <= 90.0`
- TS neighborhood stats summary:
  - average candidate pool size
  - average selected candidate count
  - bucket selection totals
  - final best feasible cost trace sample

GA must run unchanged from commit `ab02e1a` behavior, only for comparison.

## Acceptance Criteria

- No polish operators are introduced or invoked.
- TS returns feasible solutions.
- TS 300s on `R_30_10_2`, seed `44`, improves from `92.62` to below `90.0`.
- Benchmark reports whether TS and unchanged GA reach:
  - ALNS exact: `65.72`
  - ALNS+1%: `66.3772`
  - ALNS+5%: `69.006`
- If TS does not reach ALNS+5%, the output must still report first-hit as `null` and include enough bucket stats to diagnose why.

## Assumptions

- "只对 TS 进行邻域修改" means GA code stays unchanged.
- "两者达到 ALNS 相似结果" means benchmark both TS and unchanged GA, but only TS is modified.
- ALNS target remains the current recorded `R_30_10_2`, seed `44`, cost `65.72`.
- No final polish, periodic polish, or restart polish is allowed in this plan.
